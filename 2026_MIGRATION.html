Posterity Record · swyx.io

The 2026 Migration

A complete re-platforming of swyx.io — off Netlify and onto Cloudflare, onto SvelteKit 2 / Svelte 5, off the mdsvex/remark markdown toolchain, and onto an edge-cached hybrid rendering model where new posts go live with no rebuild.

June 2026 Netlify → Cloudflare SvelteKit 2 · Svelte 5 marked + Shiki GitHub Issues CMS

Why

The site renders blog posts from GitHub Issues at request time. Under the old setup, the GitHub token wasn't reliably available at runtime, so the app frequently fell back to the unauthenticated GitHub API (60 requests/hour) and broke. The markdown pipeline (mdsvex + remark/rehype) had also drifted into a pile of regex workarounds, and the toolchain was a major version behind across the board.

The 2026 rebuild fixes the root causes: authenticate properly on the edge, render markdown with a lean modern pipeline, and cache aggressively at the edge so publishing is instant and serving is O(1).

The rendering model

Pages split into two buckets. Truly static pages are prerendered at build. Everything that depends on post content is rendered on demand on Cloudflare Workers and stored in the Cache API with stale-while-revalidate, so the first request renders and the rest are served instantly from the edge.

Prerendered at build static

  • /about, /portfolio, /subscribe — rendered once from local markdown

Dynamic + edge-cached SWR

  • /, /ideas, /[slug]
  • /rss.xml, /sitemap.xml
  • /api/* (listContent, latestPosts, ideas/[slug])

Request flow

GET /some-post │ ├─ hooks.server.js → caches.default.match() │ │ │ ├─ HIT → return cached response (O(1), no GitHub call) │ └─ MISS → resolve() → SSR render │ ├─ fetch GitHub Issues (authenticated) │ ├─ marked + shiki → HTML │ └─ if Cache-Control has s-maxage: │ ctx.waitUntil(cache.put(...)) │ └─ Cache-Control: public, max-age=0, s-maxage=86400, stale-while-revalidate=604800

Instant publishing (on-demand revalidation)

Edit a GitHub Issue │ └─ GitHub webhook (issues event) │ └─ POST /api/revalidate ├─ verify HMAC SHA-256 (GH_WEBHOOK_SECRET, Web Crypto) ├─ derive slug from issue frontmatter └─ purge edge cache for: / · /ideas · /rss.xml · /sitemap.xml /{slug} · /api/ideas/{slug}.json · list endpoints (+ optional zone-wide purge via CF API)

By the numbers

60 → 5000 GitHub API requests/hr once authenticated on the edge
~800KB → 35KB largest client JS chunk (Shiki moved server-side)
0s rebuild time to publish a new post (webhook purge)
O(1) edge-cache serving for repeat requests
